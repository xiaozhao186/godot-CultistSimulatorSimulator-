## 2. 关键新功能与完成度评估

### 2.1 计时卡牌奖励逻辑升级

- **需求**：带计时条的卡牌在作为奖励出现在奖励面板中时，不再继续倒计时；只有当玩家把卡牌从奖励区拿到桌面上时，计时才开始/继续。
- **实现方式**：
  - `Card.gd` 中增加 `is_timer_active: bool` 字段，计时更新逻辑仅在该标志为真时递减 `current_lifetime`。
  - `EventPanel.gd` 中，奖励卡牌进入内部存储时，将 `is_timer_active` 置为 `false`，并隐藏/禁用交互；当卡牌从奖励区被拖出回到桌面时，重新启用计时。
- **效果**：
  - 玩家不会因为“慢拿奖励”而亏损卡牌寿命，更符合直觉。
  - 为后续更复杂的“奖励仓库”“任务链奖励”等机制预留了控制点。
- **完成度评估**：**100%**（已在实际事件奖励流程中验证逻辑正常工作）。

### 2.2 暂停状态下的交互与框选修复

- **问题背景**：早期版本中，当游戏处于暂停状态时，选择框绘制与部分交互会受到 `process_mode` 的影响，出现“暂停后无法框选”或交互不一致的问题。
- **主要改动**：
  - `GameManager.gd` 的 `_ready()` 中，将节点的 `process_mode` 设置为 `Node.PROCESS_MODE_ALWAYS`，保证全局输入与调试逻辑在暂停时仍可响应。
  - `SelectionMarquee.gd` 同样设置为 `PROCESS_MODE_ALWAYS`，并通过 `GameManager` 的 `selecting` 状态和世界坐标绘制选择矩形。
- **效果**：
  - 即便游戏暂停，玩家仍可以用右键拖拽框选卡牌，不会出现“暂停就锁死交互”的体验问题。
- **完成度评估**：**100%**（已通过实际暂停/恢复场景测试）。

### 2.3 主菜单与剧本选择

- **主菜单结构**（`MainMenu.tscn` + `MainMenu.gd`）：
  - 布局：全屏背景图 + 居中的 `VBoxContainer` 按钮列。
  - 按钮：
    - `New Game`：进入剧本选择面板。
    - `Continue`：若检测到自动存档存在则启用，否则禁用。
    - `Settings`：弹出设置面板（复用既有 `SettingsPanel.tscn`）。
    - `Exit`：退出游戏。
  - 剧本选择面板：
    - 扫描 `res://data/scenarios/` 下的 `.tres/.res` 文件。
    - 仅将脚本类型为 `ScenarioData` 的资源加入列表，展示为 `title (id)` 格式。
    - 选中后可点击 `Start Operation`，调用 `GameManager.start_scenario(_selected_scenario)`。
- **启动与场景切换流程**：
  1. 主菜单中选取剧本。
  2. `GameManager.start_scenario` 记录 `current_scenario_id`，然后切换到 `Tabletop.tscn`。
  3. 场景加载完成后，在桌面上按 `ScenarioData` 中的配置生成初始卡牌与 Verb。
- **完成度评估**：
  - 主菜单 UI 与基本逻辑：**90%**
  - 与剧本数据联动：**70%**（`ScenarioData` 结构已完整，但内容仍以示例为主，尚未完全覆盖所有设计文档）。

### 2.4 存档与读档系统（初版）

- **存档入口**：`SaveManager.gd`（Autoload 单例）
  - 存档路径：`user://saves/`，采用 `*.json` 文本格式。
  - 默认槽位名：`autosave`。
  - 对外接口：
    - `save_game(slot_name := "autosave")`
    - `load_game(slot_name := "autosave")`
    - `has_save(slot_name := "autosave") -> bool`
- **存储内容**：
  - `timestamp`：保存时间戳，便于调试与显示。
  - `scenario_id`：当前剧本 ID，对应 `GameManager.current_scenario_id`。
  - `cards`：桌面上所有未排队删除的 `Card`：
    - `id`、`pos_x`、`pos_y`、`lifetime`、`stack_count`。
  - `verbs`：桌面上所有未排队删除的 `Verb`：
    - `id`、`pos_x`、`pos_y`。
  - `global_variables`：预留字段，目前为空字典。
- **读档流程**：
  1. `SaveManager.load_game` 读取对应 JSON 文件并解析。
  2. 调用内部 `_apply_load_data(data)`，再转交 `GameManager.restore_game_state(data)`。
  3. `GameManager.restore_game_state`：
     - 切换到 `Tabletop.tscn` 并等待加载完成。
     - 清空桌面上现有 `Token`（卡牌、Verb）。
     - 按 `cards` 数组重新从 `CardDatabase` 取回 `CardData`，生成卡牌并恢复位置、寿命与堆叠信息。
     - 按 `verbs` 数组通过 `spawn_verb(verb_id)` 生成对应 Verb 并恢复位置。
- **与主菜单集成**：
  - 主菜单 `_ready` 中，通过 `SaveManager.has_save("autosave")` 控制 “Continue” 按钮的可用状态。
  - 点击 “Continue” 时，调用 `GameManager.load_game("autosave")`，从而完成读档并切换到桌面。
- **完成度评估**：
  - 核心存档/读档流程：**80%**
  - 已涵盖：剧本 ID、桌面卡牌与 Verb 的位置与基本状态。
  - 尚未涵盖：事件面板内部状态、奖励存储区、全局变量（资源消耗、战役进度等）。

### 2.5 剧本系统与示例剧本

- **剧本资源类型**：`ScenarioData.gd`
  - 字段：
    - `id: String`
    - `title: String`
    - `description: String`
    - `initial_cards: Array[CardData]`
    - `initial_verbs: Array[VerbData]`
    - `starting_events: Array[EventData]`
- **示例剧本**：`data/scenarios/factory_evacuation.tres`
  - `script_class="ScenarioData"`，脚本指向 `res://scripts/ScenarioData.gd`。
  - 目前主要填写了 ID、标题与说明文本（“Factory Evacuation”），各数组暂为空，作为演示入口。
- **与系统的关系**：
  - 剧本系统为“总剧本设置”的载体，未来所有战役/任务都可以以 `ScenarioData` 做为入口。
  - 游戏启动流程从“固定 Tabletop”扩展为“主菜单 -> 选择剧本 -> 动态加载桌面初始配置”。

## 3. 与 2026-01-26 版本的差异总结

在 1 月 26 日的报告中，项目重点在于：

- “事件系统 + 槽位系统 + 奖励系统”的整体设计与实现。
- 卡牌/行为作为核心实体的交互逻辑已经比较成熟。
- 存档系统尚未实现；起始流程较为固定（单一桌面场景）。

截至 2026-02-15，相比之下主要新增/改动点如下：

1. **全局流程层面**
   - 引入了**主菜单场景**，支持剧本选择与“继续游戏”按钮。
   - 切换到更加“战役/剧本导向”的启动流程，而不是直接进入桌面。
2. **状态持久化层面**
   - 初步上线了**存档/读档系统**，可在一定程度上恢复局面。
   - 为后续“章节推进”“多存档位”等需求打下了结构基础。
3. **细节体验层面**
   - 奖励卡牌不再在奖励区内继续倒计时，避免玩家“拿慢了就亏”的负反馈。
   - 暂停状态下的交互体验明显改善，框选与部分调试操作不再被暂停锁死。
4. **剧本/内容组织层面**
   - 使用 `ScenarioData` 抽象“剧本”，并在 `data/scenarios` 中落地具体 `.tres` 文件。
   - 为你后续基于文档（如“工厂撤离战剧本设计与系统覆盖报告”）快速配置多个剧本提供统一入口。

整体来看，项目已经从“单一系统验证阶段”进入到“**流程整合 + 可玩 Demo**”的阶段：  
系统之间的连接（主菜单 -> 剧本 -> 桌面 -> 事件 -> 奖励 -> 存档）清晰可行，后续主要工作将从“写系统”转向“填内容”和“打磨体验”。

## 4. 当前已知问题与后续建议

### 4.1 已知限制 / 技术债

1. **存档覆盖范围有限**
   - 当前仅存储：剧本 ID、卡牌/Verb 的基础状态。
   - 未涵盖：事件面板内部状态（进行中的事件、计时进度）、奖励仓库、主线战役进度、全局计数器等。
   - 读档后可能出现“事件 UI 状态与逻辑不完全一致”的情况（视具体玩法而定）。
2. **剧本内容仍以示例为主**
   - `factory_evacuation.tres` 目前尚未填充初始卡牌、Verb 与起始事件数组。
   - 尚未将“工厂撤离战”的设计文档完整映射为 `ScenarioData` + `EventData` 链条。
3. **Verb 与事件的恢复逻辑仍偏简单**
   - Verb 通过 `id` 恢复，但如果未来 Verb 的数据结构进一步复杂（如拥有更多运行时状态），需要补充更多字段在存档中。
4. **警告与小问题**
   - `GameManager.gd` 中定义了名为 `ScenarioData` 的常量，与全局类名同名，会出现 Godot 的警告（不影响运行，但可考虑重命名为 `ScenarioDataScript` 等）。
   - 部分信号如 `token_dropped`、`token_clicked` 在当前版本中声明但未显式使用，可视后续需求决定是否移除或接入。

### 4.2 建议的下一步工作

1. **扩展存档系统**
   - 将进行中的事件面板状态（当前事件 ID、剩余时间、内部卡牌等）纳入序列化。
   - 统一设计“全局变量”结构（资源、时间线、战役阶段等），并通过 `global_variables` 字段保存。
2. **完善示例剧本**
   - 以“工厂撤离战”文档为蓝本，将关键节点转化为 `ScenarioData.initial_events` + 一系列 `EventData` 链条。
   - 为示例剧本配置一套最小可玩的卡牌/Verb 集合，用于演示完整循环。
3. **主菜单与设置整合**
   - 视需求将 `project.godot` 的 `run/main_scene` 切换为 `MainMenu.tscn`，并验证与现有调试流程的兼容性。
   - 将更多全局设置（音量、速度、图形）通过 `SettingsService` + `SettingsPanel` 与主菜单联动。
4. **体验打磨**
   - 针对“继续游戏”“剧本描述”等 UI，增加适量文本提示与过渡动画。
   - 进一步优化暂停状态下的交互提示（例如显示“暂停中”标识，但依然允许框选、查看详情等无进度操作）。
