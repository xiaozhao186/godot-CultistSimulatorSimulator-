# 终极事件系统重构计划：全流程闭环方案

综合之前的讨论，我们将构建一个支持 **即时预览**、**多阶段流转**、**限时交互** 和 **手动收集** 的完整事件系统。

## 1. 核心状态机 (State Machine)

`EventPanel` 将围绕三个核心状态运行：

| 状态                    | 描述          | 核心行为                                                                                                  |
| :-------------------- | :---------- | :---------------------------------------------------------------------------------------------------- |
| **配置中 (CONFIGURING)** | 玩家放入卡牌，寻找配方 | • **即时预览**：放入卡牌立即触发 `instant_branches` 检查，若匹配则无缝切换事件界面（新标题/槽位）。• **自动填充**：尝试用上一阶段遗留的卡牌填充新槽位。          |
| **工作中 (WORKING)**     | 倒计时运行中      | • **限时交互**：大部分槽位锁定，但标记为 `interactive_during_work` 的槽位仍允许放入卡牌（用于应对突发需求）。• **计时结束**：触发结算，判定跳转下一阶段或进入收集。 |
| **收集 (COLLECTING)**   | 事件结束，战利品展示  | • **产出展示**：清空输入槽，生成输出槽，填入所有返还卡牌及奖励。• **手动回收**：玩家拿走所有卡牌后，面板重置。                                         |

## 2. 数据层改造

### `EventData.gd` (事件数据)

* 新增 `instant_branches` (Array\[EventBranchData])：

  * 用于配置阶段的即时跳转（如：放入“钥匙” -> 瞬间变为“开门”面板）。

### `EventSlotData.gd` (槽位数据)

* 新增 `interactive_during_work` (bool)：

  * 默认 `false`。若为 `true`，允许在倒计时期间操作此槽位（如：倒计时中途需要补给“燃料”）。

## 3. 逻辑层改造 (`EventPanel.gd`)

### A. 卡牌持久化系统 (Internal Storage)

解决“多阶段卡牌不丢失”问题。

* 新增 `internal_storage`：一个隐藏的容器，用于暂存跨阶段的卡牌。

* **流转逻辑**：

  1. **阶段结算**：倒计时结束时，将槽位中需返还的卡牌移入 `internal_storage`，销毁需消耗的卡牌。
  2. **新阶段初始化**：加载新事件数据后，优先从 `internal_storage` 中提取卡牌填入匹配的新槽位。
  3. **暂存显示**：未被填入的卡牌保留在 `internal_storage`，UI 上可显示为“被事件扣留的小图标”（可选）。
  4. **最终释放**：进入收集模式时，将 `internal_storage` 所有卡牌倾泻到输出槽。

### B. 完整交互流程

1. **放入**：玩家拖入卡牌 -> 触发 `instant_branches` -> 面板变身（如显示新槽位）。
2. **开始**：满足条件 -> 点击开始 -> 进入 `WORKING` 状态 -> 倒计时。
3. **中途**：若有开放槽位，玩家可补卡；若无，等待。
4. **结算**：计时结束 -> 检查 `branches`（基于当前所有卡牌属性）。

   * **情况 I：有后续事件** -> 切换至新 `EventData` -> 进入 `CONFIGURING` -> 自动填卡 -> 等待玩家操作。

   * **情况 II：无后续事件** -> 生成奖励 -> 进入 `COLLECTING` -> 显示所有战利品。
5. **结束**：玩家拿光卡牌 -> 面板重置为初始状态。

## 执行步骤

1. **数据升级**：修改 `EventData` 和 `EventSlotData` 脚本。
2. **状态机实现**：重构 `EventPanel.gd`，引入 `State` 枚举和 `internal_storage`。
3. **即时逻辑**：实现 `_check_instant_branches` 和动态 `setup`。
4. **收集逻辑**：实现 `_enter_collection_mode` 和战利品生成。
5. **验证**：测试完整的多阶段流程（如：研究 -> 突破 -> 领奖）。
