# 存档行为错乱与事件链回退修复：实施计划

## 目标
解决两个会影响剧本制作与玩家体验的核心问题：
1. **存档/读档后行为（Verb）错乱**：动词图标/标题变成别的剧本的动词，但点开面板却仍在跑原来的事件链，结算后又“变回”新的动词初始事件。
2. **事件链导向漏洞**：即时分支（instant_branches）按顺序命中后切到下一个事件，但该事件的卡槽规则无法接受当前卡牌，导致卡被弹出，并且系统不会回退尝试后续分支，造成某些分支永远无法进入（如“混乱火车站”无法进入“修复”分支）。

## 问题 1：存档/读档后 VerbData 被错误替换

### 根因（已定位）
- `GameManager` 会递归扫描整个 `res://data`，把所有 `VerbData` 以 `VerbData.id` 为 key 写进 `verb_db`，同 id 跨文件夹会**静默覆盖**，且扫描顺序不保证稳定。
- 存档中标准动词只保存了 `id`，读档时 `spawn_verb(id)` 直接从 `verb_db[id]` 取到“最后覆盖的那份 VerbData”，从而产生“图标/标题变了，但面板状态仍恢复了旧 event panel_state”的错乱现象。

### 实施方案
1. **存档增加 verb_path**
   - `Verb.serialize()` 增加字段 `verb_path`：优先写入 `self.data.resource_path`（标准动词）。
   - debug 动词保持现有 `debug_event_path` 逻辑不变。
2. **读档按 verb_path 精确恢复**
   - `GameManager.restore_game_state()`：若 `verb_path` 存在且可 load 出 `VerbData`，则按该资源创建 Verb（不再依赖 `verb_db` 的 id 映射）。
   - 兼容旧存档：没有 `verb_path` 时继续走原 `spawn_verb(id)`。
3. **扫描阶段对重复 id 给出告警**
   - `GameManager._scan_resources()` 在写入 `verb_db[id]` 前，如已存在同 key，记录 warning（输出两份资源路径），便于你后续清理/规范 id。
4. （可选增强）旧存档的“纠错回退”
   - 若旧存档没有 `verb_path` 且检测到该 `id` 在工程内存在多个 `VerbData`，尝试用 `panel_state` 内的事件 id/路径匹配 `VerbData.default_event` 来选取最合理的那一份（匹配不到再退回现状）。

### 验收
- 保存并读档后：
  - 场上动词的标题/图标不再被别的剧本动词替换；
  - 进行中的计时事件链继续正确恢复；
  - 非 auto_start 的动词不会被错误替换成“结局判断”等其他动词。

## 问题 2：即时分支不回退导致卡槽弹出（导向漏洞）

### 复现（你给的例子）
- `撤离初始.tres` 的 `instant_branches` 先命中“火车站”导向 `发车.tres`，但 `发车` 的“火车站槽” forbids `混乱`，导致“受损火车站（火车站+混乱）”被弹出。
- 但同一张卡同时满足“混乱”分支应导向 `修复.tres`，而系统不会在“下一个事件不接收当前卡牌”时回退尝试后续分支，导致永远进不去修复。

### 实施方案
1. **为 instant_branches 增加“可承载当前卡牌”的分支选择**
   - 在 `EventPanel._check_instant_branches()` 中：
     - 先收集所有条件命中的分支（保持原顺序）。
     - 对每个分支计算 `target_event`，并进行一次“卡牌可保留”的可行性检测：
       - 当前 active_slots 中的已放入卡牌是否能在 `target_event.slots` 中找到可接受的槽位组合。
     - 选择第一个“条件命中且可保留卡牌”的分支进行切换。
     - 若没有任何分支可保留卡牌，则保持当前事件不切换（避免把卡弹出去）。
2. **可行性检测实现**
   - 新增一个纯逻辑校验函数（不依赖节点实例化）：
     - 输入：`cards:Array[CardData]` + `slot_datas:Array[EventSlotData]`
     - 输出：是否存在一种分配使每张卡至少能落到一个槽（遵守 required/forbidden tags/attributes）。
   - 槽位分配用小规模回溯/匹配（slot 数通常很小），避免贪心导致误判。
3. **即时分支回退栈（instant_branch_stack）同步适配**
   - `_check_instant_branches()` 与“回退到 prev_event”的逻辑都使用同一套“可保留卡牌”筛选，避免出现频繁切换/卡牌弹出。

### 验收
- “受损火车站（火车站+混乱）”放入 `撤离初始` 后：
  - 不会切到 `发车` 导致弹出；
  - 会正确切到 `修复`（若其分支条件存在且满足）。
- 不影响其他只依赖单一条件的 instant branch 行为。
