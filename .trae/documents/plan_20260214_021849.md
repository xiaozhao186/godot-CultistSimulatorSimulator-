# Phase 5: Deep Structure Editing & Logic Correction

This plan addresses the core misunderstanding of the data structure (`Branch` -> `Outcomes` -> `Target Event`) and implements full in-node editing as requested.

## 1. Data Structure Alignment
The previous implementation assumed 1 Branch = 1 Target. The actual structure is:
`EventData` -> `branches` (Array[Branch]) -> `outcomes` (Array[Outcome]) -> `target_event`

**New Visualization Strategy:**
- **Branches** are container sections (Logic Blocks).
- **Outcomes** are the actual Output Ports (Connections).
- **Conditions** are filters for the Branch.

## 2. EventNode UI Redesign
We will rebuild `EventNode` to reflect this hierarchy.

### Hierarchy Layout:
*   **Header**: Title, Desc, Properties (Done)
*   **Instant Branches Section**:
    *   `Branch 0` (PanelContainer)
        *   **Conditions**: List of conditions + Add button.
        *   **Outcomes**: List of rows. Each row:
            *   `SpinBox` (Probability)
            *   **Output Port** (Right side) -> This is the connection point!
            *   `Delete` Button
*   **Result Branches Section**:
    *   (Same structure as above)

## 3. Implementation Details

### A. Branch & Outcome Management
-   **Add Branch**: Creates a new `EventBranchData`.
-   **Add Outcome**: Inside a branch, creates a new `EventOutcome`. This triggers a new **Output Port** on the node.
-   **Add Condition**: Inside a branch, creates a new `EventCondition`.

### B. Connection Logic (`GraphView`)
-   **Slot Mapping**: Since outcomes are dynamic and nested, we need a robust mapping system.
    -   We will iterate through all branches and all outcomes to assign slot indices sequentially.
    -   `map_slot_to_outcome(slot_index)` -> returns `{ branch: B, outcome: O }`.
-   **Connecting**:
    -   When connecting `Slot X` -> `Target Node`:
        -   Find the `Outcome` resource corresponding to `Slot X`.
        -   Set `outcome.target_event = target_node.event_data`.

### C. In-Node Probability Editing
-   Next to each Outcome port, place a `SpinBox` bound to `outcome.probability`.
-   Connecting/Disconnecting updates `target_event`, but Probability is editable anytime.

### D. In-Node Condition Editing (Advanced)
-   Replace the "Cond" button with a dynamic row:
    -   `OptionButton` (Type: HAS_TAG, SUM, etc.)
    -   `LineEdit` (Tag/Attribute)
    -   `SpinBox` (Value)
    -   `OptionButton` (Op: >, <, =)

## 4. Execution Steps

1.  **Refactor `EventNode.gd`**:
    -   Rewrite `_rebuild_slots()` to iterate Branches -> Outcomes.
    -   Render nested UI for Conditions and Outcomes.
2.  **Refactor `GraphView.gd`**:
    -   Update connection logic to look up the correct Outcome based on the new slot layout.
3.  **Persistence**:
    -   Ensure every UI change (Probability, Condition Tag, etc.) triggers `ResourceSaver.save()`.

## 5. Migration Note
This is a significant change to how the node is rendered. Old connections might break visually if the slot indices shift, but the underlying data (`target_event` references) will remain intact. Re-saving the graph will fix visual connections.
